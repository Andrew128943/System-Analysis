# System-Analysis

DriverPass is a driver education startup founded in response to more than 65% of driving test applicants failing, because they lacked structured, accessible preparation tools. The client, Liam, wanted a web-based platform that would address this gap by combining online coursework and practice testing with in person driving lessons. The system I was asked to design needed to serve four distinct user types - customers, sceretaries, the owner, and an IT officer. They each interact with the system through role-appropriate permisisons. Customers could self-register, enroll in one of three tiered training packages, schedule driving lessons and track their own test progress. Secretaries managed those appointments on behalf of customers. The owner had oversight of reporting, activity logs, and package management. The IT Officer handled all account administration.

I think the area where I did particularly well was translating the interview script into structured actionable functional requirements. Rather than writing vague requirements, I explained how that specific requirement would work. For example, with security I explained that it would use HTTPS/TLS encryption for all traffic, bcrypt password hashing, role-based access control per user type, account lockout after repeated failed login attempts, and a time-limited one-time-use password reset link. Requirements like that gives developers a deeper understanding and more to build from.

If I could go back and improve one section, it would be my UML Diagrams, specifically the UML Class Diagram. I would revisit the class relationships and cardinalities more carefully to ensure that every entity is properly represented with clearly defined attributes and associations. 

My approach to understanding user needs started with close reading of the client interview. I paid attention to not just what was explicitly requrested, buit to the why behind each request. For example, Liam didn't simply ask for a shceduling feature, he explained that DMV test failures were tied to inadequate preparation, and that structured, accountable lesson tracking was part of the solution. Consiering user needs is essential in system design because it only succeeds if it actually oslves the problem for the people using it. 

My process for this project focused on a requirements-first approach. We started by fully understanding what the system needs to do before making any decisions about how it will do it. This meant working through the business requirements document thoroughly, then I used UML as a tool for progressive refinement to see who interacts with the system and what capabilities each role needs, such as the Use Case diagram. The activity diagrams expressed specific workflows like scheduling a driving lessons, or taking a practice test. The Sequence Diagram showed real-time interaction between every major system component.
